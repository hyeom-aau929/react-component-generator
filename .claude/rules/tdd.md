# TDD Rule — React Component Generator 프로젝트

⚠️ **이 규칙은 Rigid — 상황에 맞게 변형하지 마라. 단, 프로젝트 CLAUDE.md/AGENTS.md에 TDD 관련 규칙이 있으면 그쪽이 우선한다.**

---

## 적용 기준

### TDD 반드시 적용 대상

- **비즈니스 로직**: 프롬프트 처리, 컴포넌트 생성 로직, 상태 변환
- **API 엔드포인트**: `server/index.ts` 라우팅, 요청/응답 처리, 에러 핸들링
- **유틸 함수**: `src/utils/` 내 모든 헬퍼 함수
- **Custom Hook**: 상태/사이드이펙트를 포함한 모든 훅 (`src/hooks/`)
- **버그 수정**: 재현 테스트 작성 → 수정 → 기존 테스트 유지

### TDD 불필요 대상

- **타입 정의**: `src/types/index.ts` (TypeScript 컴파일러가 검증)
- **설정 파일**: vite.config.ts, tsconfig.json, bun.toml 등
- **순수 UI 컴포넌트**: react-live 샌드박스 출력물 (SYSTEM_PROMPT 결과)
- **스타일 시트**: App.css, index.css (화면 육안 검증)
- **마이그레이션 스크립트**: 일회성 데이터 변환

## RED-GREEN-REFACTOR 사이클

### RED: 실패하는 테스트 작성

1. **한 동작 = 한 테스트**  
   예: "사용자가 프롬프트를 제출하면 API가 컴포넌트 코드를 반환한다"
   - 이 수준의 세분화. 너무 크거나 너무 작지 않아야 함

2. **반드시 실행 확인**  
   `bun run test` (또는 watch 모드) — 테스트가 **실패하는 것을 눈으로 본다**
   - 실패하지 않는 테스트는 테스트가 아니다

3. **실패 이유 명확**  
   "기능 미구현"으로 실패해야 함
   - ❌ 문법 에러, 타입 불일치 → 테스트 코드부터 수정
   - ✅ `AssertionError: expected undefined to equal "..."` ← 이 형태

4. **최소한의 테스트**  
   이 동작만 검증하고, 다른 기능은 **모킹/스텁으로 우회**하라

### GREEN: 테스트를 통과시키는 코드 작성

1. **최소한의 코드만 작성**  
   이 테스트를 통과하는 구현만
   - 하드코딩도 OK (`return "Hello"` 형태)
   - "나중에 최적화" 금지 아님 — 그냥 지금 필요한 것만

2. **YAGNI 원칙 (You Ain't Gonna Need It)**  
   지금 필요 없는 기능/예외 처리는 쓰지 마라
   - 다음 테스트가 요구하면 그때 추가

3. **전체 테스트 통과 확인**  
   `bun run test` 전체 — 신규 테스트 + 기존 테스트 모두

4. **복사-붙여넣기 중복 OK**  
   REFACTOR에서 제거한다

### REFACTOR: 코드 품질 개선

1. **목표**: 중복 제거, 변수명 개선, 헬퍼 함수 추출, 복잡도 감소
2. **GREEN 유지**: 모든 테스트가 여전히 통과해야 함  
   - 기능 변경 금지 (테스트 기반 코드 정리만)
3. **새 동작 추가 금지**  
   "이참에 다른 기능도..." → NO  
   다음 RED로 가라
4. **테스트도 refactor**: 중복된 setup/assertion 제거  
   - 하지만 테스트 의도는 명확하게 유지

### 반복

REFACTOR 다음은 다음 동작에 대한 **RED**로 돌아간다  
"한 번에 여러 동작" 금지 — 하나씩 RED-GREEN-REFACTOR 완료 후 진행

## 삭제 강제 규칙

### 프로덕션 코드를 테스트 전에 작성했다면?

**즉시 삭제** — 테스트 작성 전에 작성한 모든 코드는 제거  
RED부터 재시작하라

### "참고용으로 남겨둘 것" 금지

- 프로덕션 코드는 테스트가 명시적으로 검증하는 것만 존재
- 테스트 없으면 "낭비"다 (언제 깨질지 모르니까)
- 미사용 코드/주석은 나중에 자동 정리

## 변명 차단표

| 변명 | 반론 |
|------|------|
| "너무 단순해서 테스트 불필요" | 단순한 코드일수록 테스트가 빠르다. 단순함이 보장하지 않는 것들 (엣지 케이스, 의도)을 테스트가 명시한다. |
| "나중에 테스트 추가하겠다" | 뒤로 미루면 99% 안 한다. 지금 RED-GREEN하면 시간이 더 적게 걸린다 (버그 찾기+고치기 비용 vs 테스트 작성 비용). |
| "시간이 없다" | TDD는 **더** 빠르다. 디버깅 → 추측 → 추가 버그 사이클이 없다. 5분 테스트 vs 30분 디버깅. |
| "프로덕션 코드 삭제하면 낭비" | 테스트 없는 코드는 이미 낭비다. 언제 깨질지 모르니까. 테스트가 보호막이고, 리팩토링의 안전벨트다. |
| "프로토타입이니까 TDD 생략" | 프로토타입도 TDD다. 빠르게 신뢰할 수 있는 코드를 만드는 게 목표다. 프로토타입이 프로덕션화면? 그때 테스트가 부채가 아니라 자산이 된다. |

## 프로젝트별 규칙 우선 조항

이 파일은 **기본값(fallback)**이다.

- 만약 `CLAUDE.md`, `AGENTS.md`, `server/AGENTS.md`, `src/AGENTS.md`에 TDD 관련 규칙이 명시되어 있으면 **그쪽이 우선**한다
- 충돌 발생 시 프로젝트 규칙을 먼저 읽고, 부족한 부분만 이 파일을 참조하라
- 이 파일의 규칙과 실제 코드/테스트 간 괴리가 발생하면 프로젝트 규칙을 먼저 업데이트하라

## 참고
