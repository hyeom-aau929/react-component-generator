# 커밋 스킬

프로젝트의 변경사항을 분석하고 적절한 한국어 커밋 메시지를 생성하여 커밋하는 스킬입니다.

## 절차

### 1단계: 변경사항 분석

```bash
git status
git diff --name-only
git diff
```

현재 스테이징된 파일과 수정된 파일을 확인합니다.

### 2단계: 논리적 단위로 분류

변경사항을 다음 기준으로 분류합니다:
- **같은 기능/버그 수정** → 함께 커밋
- **다른 목적** → 분리 커밋
- **스타일, 설정** → 별도로 처리

**예시:**
- App.css + index.css 수정 (UI 스타일링) → 함께
- 폰트 변경 (index.css) vs 레이아웃 변경 (App.css) → 분리

### 3단계: 커밋 메시지 작성

**형식:**
```
[type]: [subject]

[optional body]
```

**type 선택:**
- `feat` - 새로운 기능 추가
- `fix` - 버그 수정
- `refactor` - 코드 구조 개선 (기능 변경 없음)
- `style` - 스타일 변경 (UI/CSS 개선)
- `chore` - 설정, 의존성 등 기타 변경
- `docs` - 문서 변경

**subject 작성:**
- 과거형으로 작성 (예: "추가", "변경", "수정", "개선")
- 명확하고 간결하게 (50자 이내 권장)
- 마침표 없음
- 첫 글자는 대문자

**예시:**
```
feat: Pretendard 폰트 적용 및 한글 렌더링 개선
fix: 미리보기 배경색 문제 해결
style: glassmorphism 효과 추가 및 색상 팔레트 업데이트
chore: 전역 스타일 변수 최적화
```

### 4단계: 사용자 승인 및 커밋 실행

분류된 변경사항과 제안 커밋 메시지를 사용자에게 제시합니다:

```
다음과 같이 커밋을 나누어 실행하겠습니다:

[변경 1]
파일: src/index.css, src/App.css
커밋: style: Pretendard 폰트 적용 및 한글 렌더링 개선

[변경 2]
파일: src/App.css
커밋: fix: 미리보기 배경색 문제 해결

진행하시겠습니까?
```

사용자의 승인 후:
```bash
git add [files]
git commit -m "[message]"
```

## 고려사항

- **변경사항 검토**: 커밋 전에 항상 사용자에게 변경사항을 제시하고 승인받습니다
- **스테이징**: 필요시 `git add` 또는 `git add -p`로 파일을 선택적으로 스테이징합니다
- **메시지 품질**: 메시지는 프로젝트 이력과 코드 리뷰에 도움이 되도록 명확하게 작성합니다
- **한국어 사용**: 모든 커밋 메시지는 한국어로 작성합니다 (type은 영문)
